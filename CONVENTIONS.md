# Folder & File Conventions -- Read Before Touching Anything

This is the **single source of truth** for how this repo is organized. Everything is built around the rule: *one canonical layout, one canonical script flow, no exceptions.* If you (or any future agent) ever feels tempted to create a new folder or filename pattern, update this file first, then the code, then the data -- in that order.

## CRITICAL: Large Channel Handling (Fixed 2026-05-08)

**Problem**: Asha Nayaswami channel (1,909 videos) was timing out at 600 seconds, only capturing ~291 videos.

**Solution**: `src/download_resilient.py` now uses **adaptive timeout**:

| Channel Size | Timeout | Applied To |
|---|---|---|
| < 500 videos | 600s (10 min) | Kerry K, Elu├▒a, etc. |
| 500-2000 videos | 3600s (60 min) | Pam Gregory, Teal Swan |
| 2000+ videos | 7200s (120 min) | Asha Nayaswami (1909 vids) |

**No manual intervention needed** -- downloader auto-detects and scales timeout based on estimated channel size. Logs show which timeout is active.

If a channel exceeds 2000 videos, the timeout will automatically use 120 minutes.

## 2. Naming Rules

| Thing                      | Rule                                                    | Example                            |
|----------------------------|---------------------------------------------------------|------------------------------------|
| Channel folder in `out/`   | Exact `uploader` field from YouTube, **no rewrites**.   | `Pam Gregory`, `EmilioOrtiz`       |
| Channel folder in `markdown/` | `sanitize(channel name)` -- spaces become `_`.        | `Pam_Gregory`, `Emilio_Ortiz`      |
| Per-video folder in `out/` | The 11-char YouTube video id. **Never the title.**      | `-PhDcDHK1vs/`                     |
| Markdown filename          | `<sanitized_title_truncated_80>_<video_id>.md`          | `Pluto_Unleashed_-EzLrfSslIE.md`   |
| Plain-text companion       | Same stem, `.txt` extension. Optional.                  | `Pluto_Unleashed_-EzLrfSslIE.txt`  |

Why id-based folders in `out/`? Titles change, contain emojis, slashes, and Unicode. The video id never changes, fits any filesystem, and is what the canonical DB joins on.

## 3. Per-Video Files

Inside `out/<channel>/<video_id>/`, **exactly these files**:

- `<title> [<video_id>].info.json` -- raw yt-dlp metadata. Required.
- `<title> [<video_id>].en.vtt` -- English captions (auto or manual). Required.
- `transcript.en.txt` -- cleaned plain-text transcript. Generated by `markdown_generator.py`.

Things that should **NOT** be there (legacy artifacts to clean):
- `*.en-orig.vtt`, `*.en-en.vtt` -- duplicate caption variants. Delete.
- `transcript.md` -- markdown belongs in `markdown/`, not `out/`. Delete.
- Any other `*.vtt` for non-English unless you've added a multi-language flow.

## 4. Workflow -- The Only Way to Add a Channel

```bash
cd ~/youtube_transcript_exporter
./export.sh --full-pipeline https://www.youtube.com/@SomeChannel
```

`--full-pipeline` runs, in order:

1. **Download** with yt-dlp into `out/<uploader>/<id>/...`
2. **Rebuild metadata** -- regenerates `db/canonical.json` from every `.info.json`
3. **Rebuild markdown** -- regenerates `markdown/` from `db/canonical.json`
4. **Audit** -- prints counts and flags inconsistencies

You should never run yt-dlp directly. You should never hand-edit anything in `out/` or `markdown/`. If something looks wrong, **fix the source** (`db/canonical.json` or the script), then re-run `--rebuild-markdown`.

---

## 5. The Idempotency Rule

Every step must be safely re-runnable.

- Running `--full-pipeline` on a channel you already have is fine: yt-dlp skips existing files.
- Running `--rebuild-metadata` always produces the same `canonical.json` from the same inputs.
- Running `--rebuild-markdown` wipes and regenerates `markdown/` from `canonical.json`.

If a script ever creates duplicate files, leaves old ones behind, or produces different output on a second run with the same inputs -- that's a bug. Fix the script.

## 7. Markdown File Schema

Every file in `markdown/<Channel>/` has YAML frontmatter then transcript:

```markdown
---
id: abc123
title: "Pluto Unleashed with Rosie Finn"
channel: Pam Gregory
guest: Rosie Finn
guest_confidence: 0.95
date: 2025-01-15
duration_seconds: 1234
views: 50000
likes: 1200
tags: [astrology, pluto, prediction]
url: https://youtube.com/watch?v=abc123
word_count: 4823
reading_time_minutes: 24
---

# Pluto Unleashed with Rosie Finn

(transcript text)
```

This is read directly by Obsidian (Dataview), Notion API, and any future LLM-RAG layer. **Do not change the schema** without updating `markdown_generator.py` and re-running `--rebuild-markdown`.

---

## 8. Cross-Referencing (Second-Brain Layer)

The whole point of this is cross-referencing across channels. Use these primitives:

- **Tags** -- shared topics across channels are findable via `tags:` frontmatter.
- **Guests** -- a guest who appears on multiple channels is the same person; the `guest` field is normalized.
- **Duplicates** -- the same video uploaded by multiple channels is tracked in `db.duplicates`.

Future RAG / search layers should consume `db/canonical.json` (structured) and `markdown/**/*.md` (full text), nothing else.

---

## 9. Things to Never Do

- Never create a new top-level folder without updating this doc.
- Never name a video folder by its title.
- Never write code that reads `out/` to produce user-facing output -- always go through `db/canonical.json`.
- Never put `.log` files in the repo root. They go in `archive/logs/`.
- Never commit `__pycache__/` or `.DS_Store`.
- Never hand-edit `markdown/`. It is generated.
- Never edit `db/canonical.json` by hand. It is generated.

## 10. When in Doubt

Run:
```bash
./export.sh --audit
```

It will tell you what's inconsistent. Fix the script, re-run, repeat.
